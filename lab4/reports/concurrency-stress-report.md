# Lab4 Ray 选做：并发压力测试报告

## 1. 实验目的

测试 llama-server 在不同并发度下的性能表现，分析并发请求对延迟、吞吐和稳定性的影响。

## 2. 实验环境

| 项目 | 内容 |
| :--- | :--- |
| 主机 | Linux archlinux, i7-13700H, 15GiB RAM |
| llama-server | `http://127.0.0.1:8080` (threads=8, Qwen3.5-2B Q4_K_M) |
| 测试工具 | Python asyncio + aiohttp |
| Prompt 数据集 | `batch-prompts.jsonl`（20 个 prompt，max_tokens=96） |

## 3. 测试方法

使用 Python `asyncio.Semaphore` 控制并发度，通过 HTTP POST `/completion` 向 llama-server 发送请求。测试 3 档并发度：

| 并发度 | 含义 |
| :--- | :--- |
| 1 | 串行执行，作为基准 |
| 2 | 同时 2 个请求竞争 CPU |
| 4 | 同时 4 个请求竞争 CPU |

每个并发度使用同一组 20 个 prompt，记录每个请求的总耗时、token 生成速度和成功率。

## 4. 实验结果

| 并发度 | 总耗时 (s) | 吞吐 (prompts/s) | 平均延迟 (ms) | P95 延迟 (ms) | 失败数 |
| ---: | ---: | ---: | ---: | ---: | ---: |
| 1 | 65.97 | 0.30 | 3298.6 | 3416.4 | 0 |
| 2 | 46.27 | 0.43 | 4624.9 | 4850.5 | 0 |
| 4 | **32.40** | **0.62** | 6467.3 | 6608.1 | 0 |

## 5. 结果分析

### 5.1 吞吐 vs 并发度

- 并发度从 1 提升到 2，吞吐量提升约 **43%**（0.30 → 0.43）
- 并发度从 2 提升到 4，吞吐量再提升约 **44%**（0.43 → 0.62）
- 并发度提升 4 倍，总吞吐仅提升约 2 倍，存在**收益递减**

原因：虽然请求可以并行处理，但所有并发请求共享同一物理 CPU 的缓存和内存带宽，当并发度超过物理核心承载能力时，上下文切换和缓存失效导致边际收益下降。

### 5.2 延迟 vs 并发度

- 并发度=1：平均延迟 **3298.6 ms**（基准）
- 并发度=2：平均延迟 **4624.9 ms**（+40.2%）
- 并发度=4：平均延迟 **6467.3 ms**（+96.1%）

原因：并发请求竞争 CPU 时间片，每个请求获得的计算资源减少，导致单个请求的 wall time 增加。

### 5.3 最佳平衡点

**并发度=2 是延迟-吞吐的最佳平衡点**：

- 相比串行，吞吐量提升约 43%
- 延迟增加约 40%，远低于并发度=4 的 96%
- P95 延迟（4850.5 ms）仍处于可控范围

### 5.4 稳定性

全部 60 个请求（20 prompt × 3 并发度）**零失败**，说明 llama-server 在并发场景下具有良好的请求队列和错误处理能力。

## 6. 结论

> 并发压力测试表明，llama-server 支持并发请求处理，但并发度并非越高越好。在单机 CPU 推理场景下，过高的并发会导致严重的请求延迟膨胀（并发度=4 时延迟翻倍），而吞吐量提升有限。实际部署中应根据用户可接受的延迟预算和硬件核心数，选择**并发度=2~3**作为单节点的最佳配置。
