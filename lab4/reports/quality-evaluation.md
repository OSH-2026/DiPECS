# Lab4 输出质量评估

## 评估设置

模型为 Qwen2.5-1.5B-Instruct Q4_K_M，CPU 推理参数固定为：

```text
threads=12, ctx=1024, batch=64, seed=42
```

对比项仅为 `temperature=0.2` 与 `temperature=0.8`。每个 prompt 生成一次，结果
保存在 JSONL 的 `stdout` 字段中。由于样本量小，本评估只用于发现明显差异，不能
代表统计显著的模型能力结论。

## 评分标准

每项 1 至 5 分，总分 20：

| 维度 | 说明 |
| :--- | :--- |
| 准确性 | 事实、概念和因果关系是否正确 |
| 指令遵循 | 是否回答指定任务并遵守格式要求 |
| 完整性 | 是否覆盖问题要求，是否因 token 上限截断 |
| 表达 | 结构、语言和可读性 |

## 评分结果

| Prompt | temp 0.2 | temp 0.8 | 主要观察 |
| :--- | ---: | ---: | :--- |
| 中文问答 | 11 | **13** | 两组均被截断；低温组错误声称云端推理降低网络延迟、提高隐私 |
| 三点摘要 | 10 | 10 | 两组都只改写原文，没有按要求列出三点 |
| Rust 代码解释 | 12 | **18** | 高温组完整解释读取、错误传播和 `Result`；低温组截断且改写了代码 |
| RPC 原因题 | 15 | **18** | 高温组给出网络延迟、通信开销、负载均衡；低温组的数据一致性原因较弱 |
| OS 知识题 | **11** | 10 | 两组都编造 GGUF 全称，并把 `mmap` 误写成一次性加载整个文件 |
| **平均分** | **11.8 / 20** | **13.8 / 20** | 高温组在本次固定种子样本中整体更完整 |

## 关键错误

1. GGUF 是 `llama.cpp` 生态使用的模型文件格式。两组回答分别把它解释为
   “Google 的文件格式”和“Global and Local Universal Format”，均无事实依据。
2. `mmap` 建立文件到虚拟地址空间的映射，不表示调用时一次性把整个模型装入物理
   内存。首次访问尚未驻留的页会发生缺页异常，再由内核从页缓存或存储读取。
3. 云端推理通常会增加网络依赖，数据离开本机也会引入隐私和合规风险，不能笼统
   写成降低网络延迟或提高隐私。
4. 摘要题的核心失败是指令遵循，而不是知识错误。两种温度都没有输出编号三点。

## 结论

本轮不能得出“温度越高质量越好”的一般结论。`temperature=0.8` 只在固定种子和
这 5 条 prompt 上表现更完整；温度改变采样分布，不会修复模型缺失的 OS 知识。
1.5B Q4 模型适合验证部署链路和基础问答，但涉及系统概念时必须人工核验。
