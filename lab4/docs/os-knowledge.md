# OS 知识点总结

## 总览

Lab4 表面上是在部署大模型推理框架，核心考察的是操作系统如何管理计算、内存、文件 I/O、网络通信和分布式资源。`llama.cpp`、RPC、Ray 和 Ceph 只是具体载体，真正需要解释的是系统瓶颈从哪里来，以及不同参数为什么会改变性能。

## 分项一：进程、线程与调度

`llama.cpp` 在 CPU 推理时会使用多线程并行计算。`--threads` 改变的是参与计算的工作线程数量。

需要理解：

- 线程数小于物理核心数时，CPU 可能没有被充分利用。
- 线程数超过合适范围后，上下文切换、缓存竞争和调度开销可能抵消收益。
- 超线程不等于完整物理核心，计算密集任务未必能随逻辑核心线性加速。
- 批量推理或多请求并发会让操作系统同时调度多个进程或线程，可能造成 CPU 抢占。

报告中可以结合 CPU 利用率、吞吐和延迟说明调度现象。

## 分项二：虚拟内存、mmap 与页缓存

大模型权重文件通常很大，加载模型时会触发大量文件读取和内存映射。

需要理解：

- `mmap` 将文件映射到进程虚拟地址空间，实际物理页通常按需加载。
- 首次访问模型页时可能发生缺页异常，内核从磁盘或页缓存中取数据。
- 页缓存会让第二次加载明显变快，因此测试模型加载时间时要说明缓存状态。
- `--no-mmap` 会让模型更倾向于完整读入内存，可能增加加载时间和内存压力，但在某些环境中减少运行期抖动。
- 内存不足时可能触发 swap 或 OOM，导致延迟陡增甚至进程被终止。

报告中应区分“虚拟内存占用”“常驻物理内存”和“文件页缓存”。

## 分项三：KV Cache 与上下文窗口

LLM 推理不仅占用模型权重内存，还会为上下文保存 KV cache。

需要理解：

- `--ctx-size` 越大，可保留的上下文越长。
- 上下文窗口变大通常会增加 KV cache 内存占用。
- prompt prefill 阶段需要处理输入 token，decode 阶段逐 token 生成，两者性能特征不同。
- 首 Token 延迟通常受 prompt 长度、batch、缓存和调度影响。
- 上下文过大但任务很短时，可能只增加资源占用，不带来质量收益。

这部分是解释首 Token 延迟、总延迟和内存占用的关键。

## 分项四：量化、缓存局部性与计算后端

GGUF 量化模型通过降低权重精度减少内存占用和带宽压力。

需要理解：

- Q4、Q5、Q8 等量化格式在模型大小、内存带宽、计算误差之间取舍。
- 较低比特量化通常更省内存、更容易在普通设备上运行，但可能影响输出质量。
- 推理速度不只取决于算力，也取决于内存带宽和缓存命中率。
- CPU 后端依赖 SIMD 指令、线程调度和缓存层级。
- GPU offload 可以减少 CPU 计算压力，但会引入显存容量、数据传输和后端编译问题。

报告中不要只写“Q4 更快”，要结合内存占用、tokens/s 和输出质量说明原因。

## 分项五：文件系统与 I/O 路径

模型、prompt 和日志都经过文件系统。I/O 路径会影响加载时间、测试稳定性和可复现性。

需要理解：

- 顺序读取大模型文件主要受磁盘吞吐和页缓存影响。
- 小 prompt 或小日志更容易受系统调用次数、同步写入和元数据操作影响。
- 本地 SSD、网络文件系统、CephFS 和对象存储的访问路径不同。
- 日志写入如果每条都同步落盘，延迟会高于批量缓冲写入。
- 文件命名、追加写和并发写可能引入冲突，需要设计日志格式和写入策略。

Rust 工具应优先输出结构化 JSONL，便于后续统计。

## 分项六：RPC 与网络通信

`llama.cpp` RPC 分布式推理把部分计算通过网络交给从机后端。

需要理解：

- RPC 会引入序列化、网络传输、远端排队和同步等待。
- 小任务中，网络固定开销可能大于分布式计算收益。
- 大任务中，如果计算划分合理且网络足够稳定，远端算力才可能抵消通信成本。
- 有线局域网、无线局域网和热点的延迟、抖动、带宽差异会直接影响结果。
- 异构设备会导致快节点等待慢节点，形成同步瓶颈。

报告中应把单机和 RPC 的对比写成“收益或开销分析”，而不是默认假设 RPC 必然更快。

## 分项七：分布式存储与 Ceph

Ceph 扩展任务关注的是多机推理系统中的共享存储层。

需要理解：

- Monitor 维护集群状态和 map。
- OSD 负责实际数据存储和恢复。
- Pool 定义对象组织方式、副本数和放置策略。
- CephFS 提供文件系统语义，对现有程序更透明。
- 对象存储适合用对象名组织模型、prompt 和日志，但需要显式上传下载。
- 副本数提高可靠性，但写入时需要更多网络和磁盘 I/O。
- 故障恢复会消耗后台带宽，影响前台读写性能。

Ceph 实验的重点是解释共享、可靠性和性能之间的取舍。

## 分项八：测量方法与可观测性

系统实验的数据质量取决于测量方法。

需要理解：

- 每组实验要记录硬件、系统版本、命令、模型、参数和网络环境。
- 单次结果容易受缓存、后台任务和网络抖动影响，建议重复测量。
- 平均值不能描述尾延迟，必要时记录 P95 或最大值。
- 吞吐、延迟和资源占用可能互相冲突，需要一起解释。
- 原始数据应保留，报告中的表格应可追溯。

Rust 测量工具至少应记录开始时间、结束时间、退出状态、命令参数、stdout/stderr 路径和指标摘要。

## 分项九：与 AIOS 项目的联系

DiPECS 是面向 Android/Linux 的 AIOS 原型，Lab4 可以作为“本地系统观测 + 推理后端”的补充实验。

可联系的点：

- 本地推理和远端推理的取舍，对应 AIOS 中实时性与能力上限的权衡。
- 隐私脱敏后的上下文传给 LLM，要求推理链路可观测、可控、可复现。
- 多机推理服务需要调度、存储和故障处理，和 AIOS 后端能力路由有关。
- 模型文件和日志管理影响系统部署成本，也影响实验可审计性。
