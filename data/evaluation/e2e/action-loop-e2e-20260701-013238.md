## 双轨发送与 adb forward 失真(诚实说明)

- **轨道A(daemon 真发)**走完整 Rust 转发链(`AndroidAdapter::forward`)。
  但经 `adb forward` 转发时,adb 用户态 TCP 代理在转发数据与转发 FIN 之间存在
  调度间隙:发送端 write 后立即关连接,FIN 追上尚未推送到设备的数据,Android 侧
  `accept` 后首次 read 即 EOF,读到空 payload,记 `authorized_action_socket_empty`。
  这是 adb forward 的固有失真(业界亦有记录),**仅存在于'开发机 daemon 经 adb 打
  设备 app'这条验证通道**。
- **生产部署无此问题**:daemon 终态是设备内 `/system/bin/dipecsd`(见
  `docs/src/design/daemon-architecture.md`),与 app 同机走内核 loopback,字节流
  严格有序,数据必先于 FIN 到达。
- **轨道B(取证发送器)**走与 daemon(AndroidAdapter)相同的 execute 信封,HMAC 覆盖
  freshness window 与 length-prefixed `action` 字节(canonical 串
  `dipecs.android.bridge.execute.v1`),与 `AndroidAdapter::canonical_execute_envelope_input`
  / 设备侧 `BridgeExecuteProtocol` 逐字节一致。仅在 write 后延迟再关以规避 adb 竞态,
  用于在验证通道取得'动作真执行'旁证。它不替代、不修改生产发送路径。
