## 中期汇报要点

### 阶段目标

完成"用户行为采集 → 本地脱敏整理 → 云端 Skills 判断 → 本地优化执行 → 结果记录"的最小闭环。

### 已完成的模块

#### 基础采集

- UsageStatsManager 应用使用采集
- NotificationListenerService 通知事件采集
- 基础上下文（时间、网络、电量）采集

#### 核心链路（Rust）

- `aios-spec` 事件模型 — `RawEvent`、`SanitizedEvent`、`StructuredContext` 类型体系
- `PrivacyAirGap` 隐私脱敏引擎 — RawEvent → SanitizedEvent，原始 PII 不可恢复
- `WindowAggregator` 窗口聚合器 — 10s 时间窗口，自动构建 `ContextSummary`
- `MockCloudProxy` 云端通信骨架 — 6 种信号 → 意图生成规则

#### 策略与执行

- `PolicyEngine` 策略引擎 — 风险等级 + 置信度双重校验
- `DefaultActionExecutor` 动作执行器 — 5 种动作类型骨架
- `ActionBus` 动作总线 — mpsc channel 解耦采集与处理
- 完整处理管道 — Collection → Sanitize → Aggregate → Infer → Evaluate → Execute

#### 测试

- 63 个测试全部通过
- 覆盖：脱敏 (5) / 窗口聚合 (17) / 策略引擎 (11) / 动作执行 (14) / 云端模拟 (9) / 动作总线 (7)

### 架构变更

daemon 二进制已独立为 `aios-daemon`，修正反向依赖，恢复 `spec` 作为协议中心、`core` 负责审查、`collector` 负责采集、`agent` 负责决策、`action` 负责授权动作执行的边界。

### 待解决问题

- MockCloudProxy → 真实 HTTPS 通信（reqwest + rustls）
- Kotlin → Rust JNI bridge（NotificationListenerService 事件传入 daemon）
- GoldenTrace 录制与回放接入主循环
- 真机 / 模拟器部署验证

## 会后判断

中期阶段证明了最小闭环可以跑通，但也暴露出几个问题：

- `MockCloudProxy` 适合演示，但不能代表最终推理架构。
- Android 到 Rust 的 JNI 接入成本较高，不一定是最短路径。
- 动作层虽然有 `ActionBus` 和 `DefaultActionExecutor`，但“模型建议”和“可执行动作”的边界还不够硬。
- GoldenTrace 结构已经存在，但还需要形成可重复回放和 audit hash。

这些问题成为后续 5 月到 6 月工作的主线：重构决策路由、先用 JSONL 打通 Android production ingress、收紧 AuthorizedAction 和动作审计、补齐 replay/audit。
