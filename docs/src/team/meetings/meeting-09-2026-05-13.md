# 第 9 次会议纪要：中期后架构重构

## 摘要

本次会议复盘中期原型，决定在继续扩展前先收紧架构边界。讨论重点包括 `aios-spec` 的协议中心地位、Android 采集入口、`DecisionRouter`、`PolicyEngine` 和动作执行之间的职责划分。

## 会议信息

- **日期**：2026-05-13
- **议题**：中期后架构重构与边界收紧
- **阶段**：中期后重构
- **参会人**：lsy, srh, ljs, wxz, sdd

## 背景

中期版本已经跑通了 `RawEvent -> PrivacyAirGap -> StructuredContext -> MockCloudProxy -> PolicyEngine -> ActionExecutor` 的最小闭环，但继续扩展时出现了边界混杂的问题：

- Android 采集、daemon 采集和 replay 输入需要统一入口协议。
- `MockCloudProxy` 的职责不清，容易把本地规则、云端推理和动作建议混在一起。
- 动作执行层还没有足够强的授权边界。
- 早期 “Skills 自迭代” 方案过于依赖云端，不利于默认本地优先。

## 讨论内容

### 1. `aios-spec` 作为协议中心

会议决定继续强化 `aios-spec` 的 SSOT 角色。跨模块共享的数据结构必须先在 `aios-spec` 中定义，再由 collector、core、agent、action 和 CLI 使用。

需要稳定下来的核心类型包括：

- `RawEvent`
- `CollectorEnvelope`
- `SanitizedEvent`
- `StructuredContext`
- `IntentBatch`
- `CapabilityLevel`
- `AuthorizedAction`

### 2. 分层采集与决策路由

大家认为中期后最重要的架构动作是把“推理接口”拆清楚。

后续方向：

- Android App 只负责采集 public API 信号，不直接生成最终上下文。
- `aios-collector` 负责 ingress 和 normalize。
- `aios-core` 负责 Privacy Air-Gap、窗口聚合和策略审查。
- `aios-agent` 只接收 `StructuredContext`，只输出 `IntentBatch`。
- `aios-action` 只执行经过授权的动作。

这次讨论后来沉淀为 [RFC-0001](../../rfc/0001-layered-collection-and-decision-routing.md)。

### 3. `DecisionRouter` 初步方案

会议决定替换单一 `MockCloudProxy` 的思路，引入 `DecisionRouter`：

- `RuleBasedBackend`：默认本地规则，处理确定性、低风险、高频场景。
- `LocalEvaluatorBackend`：本地评估或轻量分类器，处理低风险候选和是否升级。
- `CloudLlmBackend`：只在配置完整且需要语义理解时启用。
- `FallbackNoOpBackend`：失败、熔断或上下文不足时回落。

这样可以避免所有窗口都走云端，也让系统在没有 API key、没有网络或云端失败时仍可运行。

### 4. 动作边界初步收紧

会议认为“模型建议动作”和“系统可执行动作”必须分开。后续实现需要避免 action 层直接接收模型输出。

初步约定：

- 推理后端只能输出 `IntentBatch`。
- `PolicyEngine` 对每个建议动作做风险、置信度和能力检查。
- 只有策略通过后，动作才能进入执行层。
- 执行层需要保留 audit record，方便 replay 和复盘。

## 决议

- 采用分层采集和决策路由方案。
- 开始编写 RFC-0001，明确 collector、core、agent 和 action 的边界。
- 默认路径从“云端 Skills 主导”调整为“本地规则优先，云端可选增强”。
- 后续动作治理需要引入更强的授权对象，而不是直接执行 `SuggestedAction`。
