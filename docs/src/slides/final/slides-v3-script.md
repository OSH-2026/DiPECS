# DiPECS v3 演讲稿

目标时长：约 38 分钟；剩余约 2 分钟用于换页、停顿或老师临时提问。以下编号与 `slides-v3.md` 一致。

## 1. 封面（0:00—0:40）

大家好，我们汇报的项目是 DiPECS。它是一个面向 Android 和 Linux 的本地优先 AIOS 原型系统，探索智能操作系统中的感知、决策、授权执行与审计闭环。我们关注的问题不是让模型直接控制操作系统，而是如何让智能决策在本地隐私、权限、策略和失败语义的约束下参与资源管理。

## 2. 一句话理解（0:40—1:30）

整个系统可以概括为六步：感知 Android 和 Linux 状态，经 Privacy Air-Gap 脱敏，聚合为结构化上下文，由本地规则、本地评分或可选 LLM 生成意图，再经过策略授权和生命周期执行，最终形成可回放审计。这里最重要的设计是：模型输出只是候选建议，真正的系统动作必须有限、可拒绝、有唯一终态。

## 3. 汇报路线（1:30—2:00）

接下来先解释项目解决的问题和 OS 定位，然后介绍系统机制。第三部分重点讲隐私、决策和动作治理。由于这次不做容易受环境影响的现场 Demo，我们使用已经保存的事件、裁决和设备终态作为静态证据。最后介绍实验结论及其边界。

## 4. 章节页：问题、目标与定位（2:00—2:10）

首先看我们为什么要做预测式资源管理。

## 5. 被动响应与有限预测（2:10—3:20）

AIOS 面临的不只是模型能力问题。应用切换、通知、进程和设备信号分散在不同 OS 接口；通知正文和行为数据具有隐私风险；模型输出本身不具备系统权限；网络与模型失败也不能阻塞控制路径。DiPECS 研究如何把这些信号组织为本地上下文，并把模型建议限制在授权、策略和审计边界内。预测一定会错，因此低置信度时最正确的动作就是 NoOp。

## 6. 典型场景（3:20—4:20）

以附件通知为例。用户收到聊天应用的文件通知，系统不需要知道正文，只需保留“来自哪个应用”和“包含文件语义”。本地决策器可以提出低风险资源准备；策略检查目标确实在当前上下文、动作风险低、能力允许，才授权执行，否则安全地 NoOp。这个例子同时包含三个约束：正文不进入模型，模型不能直接执行，预测不可靠时不消耗资源。

## 7. 目标与非目标（4:20—5:20）

我们要验证的是一条完整可信路径，而不是宣称已经改造 Android 内核。当前项目不修改 scheduler 或 LMKD；合成下一应用准确率不能外推到真实用户；部分跨应用动作只有系统签名环境才完整可用。我们还明确区分模拟器实测、离线回放、合成数据和估算值，避免把工程原型包装成不存在的生产能力。

## 8. 三项贡献（5:20—6:20）

贡献有三项。第一，把 Android 和 Linux 的异构信号转成隐私受控的本地上下文；第二，设计本地优先、可选 LLM、可熔断降级的分级决策；第三，把候选动作置于能力、策略、认证和生命周期之下，形成可回放审计。贯穿这些设计的是 OS 中常见的机制与策略分离、最小权限、失败关闭和可追责原则。

## 9. 章节页：架构与 OS 机制（6:20—6:30）

下面进入系统架构。

## 10. 总体架构（6:30—7:40）

左侧 Android Collector 是感知机制层，使用公开系统 API。中间 Rust 数据面负责 schema ingress、Privacy Air-Gap 和十秒窗口。右侧 AIOS 控制面由 DecisionRouter、PolicyEngine 和 ActionLifecycle 组成。模块之间有三类持久边界：JSONL 用于增量输入与 replay，localhost socket 用于授权动作，NDJSON audit 用于记录终态。云模型不是可信计算基的一部分，任何一层都不单独拥有原始数据、决策和执行三种全部权限。

## 11. 部署和进程边界（7:40—8:50）

Android App 负责它最擅长的公开 API 采集和设备侧执行；`dipecsd` 负责系统状态、脱敏和控制逻辑；云端完全可选，只能看到脱敏后的上下文，且没有设备权限。云端失败不会阻塞系统，因为 Router 会回退到本地。生产目标是 daemon 与 App 在同一设备上运行，通过内核 loopback 通信。

## 12. OS 相关性（8:50—10:00）

这个项目和传统 OS 的关系分三层。接口层使用 `/proc`、进程、文件和 socket；问题层处理内存、I/O、IPC 和进程生命周期；设计原则层实现机制策略分离、能力控制和审计。我们准确的定位是本地优先的用户态 AIOS 原型：用 OS 机制约束智能决策，而不是修改内核算法。

## 13. `/proc` 进程监控（10:00—11:10）

Rust 的 ProcReader 扫描 `/proc/<pid>/status` 和 `oom_score`，提取 RSS、Swap、线程数和 UID。每轮生成 pid 到快照的映射，再与上一轮比较，只为变化进程产生事件。这样把高频轮询转化为稀疏事件流。规则层还会根据高 RSS 或高 Swap 识别内存压力候选。这是典型的内核状态经 procfs 暴露到用户态的方式。

## 14. Daemon 模型（11:10—12:20）

`dipecsd` 使用 fork 创建子进程，父进程退出，子进程通过 setsid 脱离原会话，并将标准文件描述符重定向到 `/dev/null`。运行时有采集与处理两个 Tokio 任务，通过有界 channel 解耦。SIGINT 和 SIGTERM 通过 broadcast 通知各任务退出，处理任务在停止前刷新剩余窗口。开发阶段也可以通过 `--no-daemon` 前台运行。

## 15. Android OS 服务（12:20—13:30）

Android 侧没有绕开权限模型。UsageStats 需要 Usage Access，通知监听和无障碍需要用户显式启用，前台服务持续展示通知，维护任务交给 JobScheduler。选择公开 API 的代价是信号能力弱于内核 hook，但优势是权限边界清晰、可复现，而且普通模拟器就能验证核心链路。

## 16. Append-only JSONL（13:30—14:40）

Android 写端把每个事件作为一行追加到 JSONL，并在落盘前完成敏感字段脱敏。Rust Tailer 保存 offset，只读取新增字节；遇到末尾半行会等待下一次补齐，遇到文件截断或轮转会重置状态。schema 不兼容的事件在 ingress 被拒绝。这套设计让在线输入、离线 replay、回归测试和审计可以使用同一数据格式。

## 17. 章节页：隐私、决策与治理（14:40—14:50）

下面介绍项目最核心的可信控制路径。

## 18. Privacy Air Gap（14:50—16:00）

边界左侧的 RawEvent 可能含联系人、通知正文和会话标识，只能短暂存在于本地脱敏路径。边界右侧只保留文本长度、文字类别和 FileMention 等语义 hint，group key 直接丢弃。模型、上下文和审计代码只依赖 SanitizedEvent 类型。也就是说，这不是在输出日志时再遮盖，而是在类型和管线层面切断原始隐私的下游可达性。

## 19. 十秒窗口（16:00—16:50）

单一事件通常不能代表用户意图，所以系统按十秒聚合。窗口中可同时出现前台应用、通知语义、屏幕状态、电量以及进程资源。关闭窗口后形成 StructuredContext，并只触发一次决策。这个设计既提供关联信息，也避免每个噪声事件都调用模型。

## 20. DecisionRouter（16:50—18:10）

Router 首先检查熔断。如果短时间连续失败，就只允许 FallbackNoOp。随后检查隐私敏感度，敏感窗口强制走规则。低复杂度和明确本地动作信号走规则或 LocalEvaluator；只有复杂且允许云端时才调用 Cloud LLM。云端失败会立即使用规则结果代替，并把错误写进审计。这使网络和模型不成为关键控制回路的单点故障。

## 21. LocalEvaluator（18:10—19:20）

本地评分器不是黑盒模型，而是确定性特征组合。前台匹配、通知与文件活动的时间相关性、重复包名和行为记忆提高分数；低电量、蜂窝或离线网络、近期策略拒绝和执行失败降低分数。每窗口最多五个 intent，分数裁剪到零到一，历史反馈只能在有限范围调整。因此它快、可解释、可复现。

## 22. PolicyEngine（19:20—20:50）

Policy 是独立于模型的硬门。首先检查 schema 和后端 capability，再检查风险与最低置信度，之后验证动作类型、紧迫度和数量。关键规则是 target-in-context：模型只能操作当前窗口真正观察到的包或资源，不能凭空写一个高置信度目标。不同拒绝原因是结构化枚举，因此审计和实验不需要解析字符串。

## 23. ActionLifecycle（20:50—22:00）

每个 intent-action 坐标都会形成 ActionProposal，并且最终恰好产生一条终态记录。只有生命周期内部能够把 proposal 封装成 AuthorizedAction，因此模型和普通调用方拿不到授权构造能力。设备连接拒绝、超时、拒绝响应和非法 JSON 都进入 Failed，而不是把一次 socket write 错报为 Succeeded。这一点保证审计语义诚实。

## 24. HMAC 动作桥（22:00—23:20）

动作请求包含签发时间、六十秒有效期、canonical action JSON 和 HMAC-SHA256。认证输入同时绑定协议版本、时间窗口、动作长度和动作字节，所以截获一个标签后既不能替换动作，也不能过期重放。服务只绑定 loopback，并限制负载、超时、并发队列和连续认证失败。这相当于在进程间建立一个窄能力通道。

## 25. 动作和权限边界（23:20—24:50）

四类动作的能力不同。Prefetch 可以在普通 APK 的可访问范围内实现；KeepAlive 可以使用 JobScheduler fallback，但写 OOM 和 cgroup 通常需要特权；跨应用预热受后台启动限制；跨应用清缓存和全局 page cache 回收需要系统权限。因此同一动作可能成功、退化或明确失败。我们不会把普通 APK 无法获得的权限隐藏成成功结果。

## 26. 章节页：静态证据（24:50—25:00）

接下来用保存下来的运行记录代替现场演示。

## 27. 案例输入（25:00—25:50）

一条事件先以 CollectorEnvelope 进入 schema ingress，随后成为 SanitizedEvent，再被聚合进 StructuredContext。模拟器 E2E 使用 Android 35，证据标签是 emulator-measured、non-synthetic，而不使用容易误解为真机的 REAL。最终生成稳定 audit hash；hash 的用途不是安全签名，而是发现 replay 处理路径发生变化。

## 28. 允许与拒绝（25:50—26:50）

左侧案例中目标包确实在上下文出现，风险低、置信度达到阈值，因此批准预热。右侧即使置信度更高，但目标包没有被观察到，仍以 TargetNotInContext 拒绝。这说明模型自信程度不能代替系统事实，也不能绕过 policy。

## 29. 四动作设备证据（26:50—28:00）

设备侧对 KeepAlive、ReleaseMemory、PreWarmProcess 和 PrefetchFile 四类处理器都留下终态事件。这里要区分能力边界：PreWarm 验证的是 `own:warmup`，不等价于第三方进程预热；Prefetch 的 1.1 毫秒只表示任务入队，真正下载成功由异步 `prefetch_succeeded` 证明。设备证据经取证发送器通过 adb forward 获得；Rust 生产发送路径另有 mock-socket E2E，因此我们只主张设备处理器覆盖，不冒充 daemon 已在设备内生产部署。

## 30. 章节页：实验（28:00—28:10）

下面集中说明实验结论。

## 31. 实验问题与证据层级（28:10—29:10）

我们按六个研究问题组织数据，分别验证闭环、隐私、决策延迟、合成下一应用预测、常驻开销和治理覆盖。不同数据有不同证据强度：真实云 API、Android 模拟器、离线 replay、合成 trace 和估算值必须明确标注，不能混合成同一等级的结论。

## 32. 合成预测评测（29:10—30:30）

我们从三个确定性合成场景构造十秒上下文窗口，并把之后三十秒内第一个不同的前台应用作为标签。946 个窗口中，764 个存在未来切换，但只有 178 个切换目标已经在当前窗口的前台或通知候选中出现，因此 context-supported coverage 只有 23.3%。在这 178 个 eligible 窗口上，RuleBased Top-1 为 61.2%，LocalEvaluator 为 43.8%。这验证了评测框架和当前策略行为，但不能代表真实用户泛化能力；RuleBased 在实际给出预测时仍有 34.7%的条件错误率。

## 33. 决策延迟（30:30—31:50）

规则和 LocalEvaluator 在亚毫秒级完成，真实云 API 十轮实验 p50 约 7.3 秒、p95 约 10 秒。该实验支持的工程判断只有一个：即时控制路径必须本地，云端只能作为非即时、可选分析后端。延迟数据本身不能证明云端决策质量，因此报告不把云模型作为系统价值来源。

## 34. 隐私与治理（31:50—33:00）

对照实验中，直接构造 naive prompt 会包含 22 个原始通知片段，而 DiPECS 模型输入和审计中为零；输入也从约 63 KB 压缩到 645 字节。PolicyEngine 二十项测试覆盖风险、能力、置信度、目标、批量上限和 fallback 隔离。这里的价值不仅是测试通过，而是隐私和治理都形成了独立于模型行为的确定性边界。

## 35. 资源和吞吐（33:00—34:10）

模拟器三十次采样中，observe-only 相对基线增加约 3.6 MB PSS，action loop 增加约 5.6 MB。离线处理 2400 行 trace、1631 个有效事件耗时 128 毫秒，峰值 RSS 10.77 MB。CPU 采样受粒度影响，电池和温度是基于模拟器数据的估算，因此我们不把它们作为真实功耗结论。

## 36. 负面结果（34:10—35:30）

这一页主动说明没有被证明的内容。旧启动实验把 cold start 与 service 已启动后的 warm process 混在一起，因此“预热快 43.8%”不能作为因果结论。当前数据也没有显示 jank 改善；ReleaseMemory 后 PSS 反而增加 0.331 MB。合成预测只能描述上下文可观察的确定性场景，不能替代真实用户 ground truth。将这些负面结果明确展示，是为了让结论强度与证据一致。

## 37. 章节页：总结（35:30—35:40）

最后总结项目边界和后续方向。

## 38. 用户态控制平面边界（35:40—36:40）

项目已经使用进程、文件、IPC 和 Android 服务接口形成感知、决策、授权执行和审计闭环，并实现权限分离和安全降级。但它没有修改调度器、LMKD、VFS 或 Binder driver，部分探针和强系统动作仍需要特权部署。因此我们的准确结论是：这是一个遵循 OS 原则的本地优先用户态 AIOS 原型。

## 39. 下一步（36:40—37:30）

后续工作分三条。评测上构建匿名化真实用户 ground truth，报告 Top-1、Top-3、eligible coverage 和错误预热率，并使用独立 controller 重做启动实验。OS 集成上接入 LMKD 或 cgroup，改用 Binder 或 Unix domain socket。决策上接入端侧轻量模型，与规则基线公平对照，并加入资源预算和预热撤销机制。

## 40. 总结（37:30—38:30）

最后有三点。第一，DiPECS 已经让 Android 和 Linux 信号进入统一、可回放的本地 AIOS 闭环。第二，原始隐私和系统执行权限都没有交给模型，动作必须经过本地硬边界。第三，我们同时报告合成预测结果、设备动作终态、运行开销和负面结果，不把证据外推。DiPECS 提供的是一种机制策略分离、可降级、可回放、可审计的 AIOS 实践方案。

谢谢大家。建议主讲人停在这一页，不继续补充。若无人立即提问，可邀请四类问题：为什么不是纯规则、预测错误如何控制代价、普通 APK 的权限边界，以及项目与传统 OS 课程的关系。

## 备用问答要点

### 为什么不直接使用纯规则？

规则适合确定、低风险信号；LocalEvaluator 可以组合多信号和历史反馈；云端只用于复杂语义。Router 的价值在于按风险和复杂度选择最低成本后端，而不是强制使用 LLM。

### 预测错误怎么办？

低置信度拒绝、每批动作上限、能力白名单、target-in-context、低风险动作集合以及 NoOp 共同限制代价。未来还需要明确资源预算和撤销策略。

### 为什么用 TCP 而不是 Binder？

TCP loopback 便于 Rust 和 Android 跨语言验证，并提供明确请求/响应语义；生产化时 Binder 或 Unix domain socket 在身份认证、Android 集成和开销上更自然，这是后续方向。

### 为什么项目算 OS？

它研究的是进程、内存、I/O 和生命周期资源管理，使用 `/proc`、daemon、文件偏移、IPC 和 Android 系统服务，并应用机制策略分离、最小权限和 fail-closed 原则。但它是用户态控制平面，不是内核实现。

### 为什么删除了“预热快 43.8%”结论？

旧脚本在对照组中 force-stop 应用，而实验组先启动 collector service，再测 Activity，混入了 cold process 与 warm process 的差异；预热发送顺序也不能形成独立因果变量。因此该结果只说明已有进程时启动更快，不能证明 PreWarmProcess 的收益。正式实验需要独立 controller、独立目标应用、随机化顺序和置信区间。
